# Пояснительная записка к проекту: Разработка прототипа системы телеметрии для спортивной медицины (нагрузка, восстановление, ЧСС).

## Введение

**Актуальность проблемы:** В современной спортивной медицине и профессиональном тренировочном процессе критически важную роль играет оперативный мониторинг функционального состояния организма спортсмена. Одним из наиболее информативных показателей реакций сердечно-сосудистой системы на физическую нагрузку является частота сердечных сокращений (ЧСС). Непрерывный контроль пульса в реальном времени, а также автоматизированный расчет уровней физиологической нагрузки и стадий восстановления позволяют спортивным врачам и тренерам своевременно корректировать интенсивность тренировок, предотвращать перетренированность и минимизировать риски развития патологических изменений в миокарде. Разработка программного обеспечения для агрегации, визуализации и локального хранения телеметрических данных является актуальной задачей, направленной на автоматизацию и повышение безопасности спортивной индустрии.

В связи с этим была сформулирована цель курсового проекта:  
Разработать **прототип информационной системы телеметрии для спортивной медицины** для **автоматизированного мониторинга, анализа и архивации показателей частоты сердечных сокращений, уровня физиологической нагрузки и резерва восстановления спортсменов в реальном времени**.

Для достижения поставленной цели необходимо решить следующие задачи:  
1. Определить сценарии использования программного средства в условиях тренировочного процесса и клинического анализа.
2. Спроектировать архитектуру программного обеспечения, интерфейс пользователя и реляционную модель базы данных для хранения сессий.
3. Реализовать и протестировать прототип системы с использованием современных технологий объектно-ориентированного программирования и баз данных.
4. Разработать сопутствующую техническую документацию и пользовательские руководства.

## 1. Функциональное назначение программного средства

### 1.1 Классы решаемых задач  
Программное средство относится к классу информационно-аналитических медицинских систем реального времени и решает следующие классы задач по бизнес-целям:
* **Задачи поиска**: Обеспечение возможности быстрого нахождения архивных записей тренировочных и диагностических сессий конкретных спортсменов по их ФИО в историческом журнале.
* **Задачи анализа (проверки и валидации)**: Непрерывная валидация входящих телеметрических данных (ЧСС), автоматическая классификация текущего пульса по пяти терапевтическим зонам интенсивности (Покой, Жиросжигающая, Аэробная, Анаэробная, Максимальная нагрузка), а также расчет интегральных показателей физиологической нагрузки и остаточного резерва восстановления.
* **Задачи оптимизации трудозатрат при решении задач поиска и анализа**: Автоматизация рутинных процессов регистрации данных, мгновенное построение интерактивных графиков, автоматическое вычисление средних и максимальных значений ЧСС за сессию и их сохранение в базе данных.

Концепция программы представлена в структуре проекта. Анализ прототипов (сравнение с существующими коммерческими системами) приведён в таблице 1.

### 1.2 Функциональное назначение  
Программа предназначена для решения задач автоматизации медицинского контроля в спортивных секциях, диагностических центрах и на тренировочных сборах.

Пользователь (спортивный врач или тренер) имеет возможность:
* Вводить личные данные спортсмена (ФИО) в текстовую форму перед началом тестирования;
* Запускать сессию мониторинга телеметрии в реальном времени кнопкой «Старт»;
* Визуализировать динамику ЧСС на интерактивном графике со скальзящим временным окном;
* Проанализировать в реальном времени текущую зону интенсивности на основе цветовой индикации и текстовых подсказок;
* Осуществить остановку записи сессии кнопкой «Завершить и сохранить» с автоматическим расчетом статистических показателей (средний пульс, пиковый пульс, итоговая преобладающая зона);
* Получить результаты анализа и обработки данных в виде структурированных записей локального журнала сессий;
* Построить исторический график любой выбранной из архива тренировки для ретроспективного анализа.

### 1.3 Нефункциональные требования, ограничения и интерфейсы  
* **Требования к надежности**: Система корректно обрабатывает аномальные всплески пульса и исключает утечки оперативной памяти при длительной отрисовке графика за счет автоматического удаления устаревших узлов данных (сдвиг окна в 30 секунд).
* **Ограничения интерфейса**: Графический интерфейс является адаптивным, снабжен контейнером панели прокрутки (`ScrollPane`) для правого блока медицинских индикаторов, чтобы исключить скрытие или наложение текстовой информации при изменении размеров окна пользователем.
* **Требования к СУБД**: Использование легковесной встраиваемой СУБД SQLite, не требующей развертывания и администрирования отдельного сервера баз данных.
* **Технические ограничения**: Запуск в среде исполнения Java Runtime Environment (JRE) версии 17 или выше, с поддержкой модулей JavaFX 21.

## 2. Проектная часть

### 2.1. Архитектура приложения  
Для обеспечения **модульности кода, простоты сопровождения, читаемости и удобства масштабирования проекта** была выбрана архитектура **MVC (Model-View-Controller)** совместно с принципами распределения обязанностей по слоям (Data Access Layer / Service Layer).

Разделение системы на независимые компоненты позволяет изменять графическое представление (представление) без необходимости переписывать алгоритмы расчета зон (модель) или логику работы со встроенной базой данных (сервисный слой).

Основными компонентами приложения являются:
1. **Слой представления (View)**: Файл разметки `main_view.fxml` (определяет структуру UI на базе JavaFX) и файл `styles.css` (задает цветовую гамму, шрифты и стили элементов управления).
2. **Слой контроллеров (Controller)**: Класс `TelemetryController.java`, управляющий захватом событий от кнопок интерфейса, координирующий запуск таймеров симуляции (`Timeline`) и обновляющий элементы интерфейса на основе сигналов от модели.
3. **Слой модели данных (Model)**: Класс `TelemetryModel.java` (или внутренние структуры состояния), содержащий свойства с поддержкой механизма связывания (Data Binding) для автоматического обновления GUI.
4. **Слой доступа к данным (Database Service)**: Класс `DatabaseService.java`, инкапсулирующий SQL-запросы к СУБД SQLite для создания локальных таблиц, вставки новых сессий и выборки исторического журнала.

### 2.2. Проектирование пользовательского интерфейса  
В соответствии с ТЗ интерфейс пользователя реализован в виде многовкладочной графической формы ввода, управления и визуализации данных (двухпанельный Dashboard). 
* Первая вкладка («Мониторинг в реальном времени») содержит верхнюю панель управления сессией, центральное поле с компонентом `LineChart` и правую аналитическую панель, помещенную в `ScrollPane` для обеспечения вертикальной прокрутки карточек зон ЧСС и прогресс-баров нагрузки (`loadBar`) и резерва восстановления (`recoveryBar`).
* Вторая вкладка («Архив и Воспроизведение Результатов») спроектирована на базе разделяемой панели `SplitPane`, содержащей слева таблицу сессий `TableView` со встроенным текстовым полем поиска, а справа — исторический график `LineChart` для детального разбора выбранной записи.

### 2.3. Модель данных ПС  
Основными сущностями в системе являются:
1. **Сессия телеметрии (TelemetrySession)** — запись о проведенном тестировании/тренировке спортсмена.
2. **Точка телеметрии (TelemetryDataPoint)** — атомарное измерение показателей в конкретный момент времени.

Для их описания используются следующие основные атрибуты сущностей в таблице `sessions`:
* `id` (INTEGER, Primary Key) — уникальный идентификатор сессии в базе данных;
* `sportsman_name` (TEXT) — ФИО спортсмена;
* `duration_seconds` (INTEGER) — общая длительность сессии в секундах;
* `avg_hr` (INTEGER) — средняя частота сердечных сокращений за сессию;
* `max_hr` (INTEGER) — максимальный (пиковый) пульс, зафиксированный за сессию;
* `summary_zone` (TEXT) — результирующая преобладающая терапевтическая зона ЧСС.

### 2.4. Моделирование работы приложения  
1. *Сценарий старта*: Пользователь вводит имя -> Нажимает «Старт» -> `TelemetryController` инициализирует внутренний таймер `Timeline` -> `Timeline` циклически генерирует ЧСС -> Изменяет свойства модели -> Модель обновляет `LineChart`, `ProgressBar` и текстовые метки на форме через механизм JavaFX Binding.
2. *Сценарий сохранения*: Пользователь нажимает «Завершить и сохранить» -> `TelemetryController` останавливает `Timeline`, вычисляет `avg_hr` и `max_hr` -> Передает объект в `DatabaseService` -> Вызывается SQL-команда `INSERT INTO sessions` -> Таблица `historyTable` автоматически обновляет свое содержимое из БД.

## 3. Конструирование программного продукта

В качестве среды разработки программного продукта была выбрана интегрированная среда разработки программного обеспечения **IntelliJ IDEA 2025.3.3**. При разработке программного продукта были использованы следующие языки программирования и разметки:
* **Java** (версия 23 / 17 LTS) — реализация бизнес-логики, математических расчетов, контроллеров и моделей данных;
* **XML / FXML** — декларативная разметка графического интерфейса пользователя и привязка обработчиков событий;
* **CSS** — стилевое оформление, кастомизация визуальных компонентов и цветовое кодирование медицинских зон.

Управление зависимостями и сборка осуществляются автоматизированным инструментом **Maven** (файл `pom.xml`).

### 3.1 Описание классов  
* **`com.sports.telemetry.MainApp`**: Точка входа в приложение, наследует `javafx.application.Application`. Переопределяет метод `start(Stage stage)`, загружает FXML-ресурс, инициализирует сцену размером 1180x750 пикселей (оптимально для предотвращения сжатия UI) и устанавливает заголовок графического окна.
* **`com.sports.telemetry.TelemetryController`**: Центральный управляющий класс бизнес-логики. Содержит ссылки на UI-элементы (`@FXML LineChart hrChart`, `@FXML TableView historyTable`, `@FXML ProgressBar loadBar` и т.д.). Отвечает за обработку методов `onStartButtonClick()`, `onStopButtonClick()` и фильтрацию таблицы через `searchField`.
* **`com.sports.telemetry.DatabaseService`**: Отвечает за персистентность данных. Содержит методы создания базы данных, вставки новых сессий и выборки исторического журнала. Использует JDBC-драйвер для подключения к файлу базы данных `jdbc:sqlite:sports_telemetry.db`.

Тестированию подвергались изолированные методы на предмет корректного вычисления средних значений и граничных условий зон (переход пульса через отметки 70, 120, 150 и 170 BPM).
